# nkg.proxmox.hetzner_image

Provision Proxmox VE onto a **Hetzner dedicated server** that has been booted
into **RESCUE MODE**, using the QEMU autoinstall flow. The role builds an
unattended Proxmox autoinstall ISO, installs it onto the bare-metal disks under
QEMU, configures the installed system over SSH, then reboots into the real
Proxmox and verifies the API.

> **DESTRUCTIVE.** This role wipes the disks listed in `hetzner_image_disks`.
> Only run it against a server you intend to (re)image.

## How it works

The server must first be booted into Hetzner rescue mode manually via the
Hetzner Robot web UI (Server → Rescue → activate Linux 64-bit, then Reset →
hardware reset, wait ~2 minutes, verify `ssh root@<server-ip>`).

The role then runs five phases (`tasks/main.yml` imports each):

1. **facts** — gather/compute network facts (interface, IPv4/IPv6, private
   subnet), detect a prior install (`/usr/bin/pvesh`), and verify rescue mode.
2. **prereqs** — add the Proxmox repo, install
   `proxmox-auto-install-assistant` / `qemu` / `ovmf` / etc., download the
   latest Proxmox ISO, and build the autoinstall ISO from `answer.toml` +
   `first-boot.sh`.
3. **install** — boot the autoinstall ISO under QEMU to install Proxmox onto
   the bare-metal disks (10-20 minutes).
4. **configure** — boot the installed system under QEMU with SSH
   port-forwarding, then over SSH set hostname / `/etc/network/interfaces` /
   `/etc/hosts` / sysctl, create the admin user, deploy SSH keys, and power off.
5. **post** — clean up temp files, reboot into the installed Proxmox, and wait
   for the API on port 8006.

Phases 2-4 are skipped automatically when Proxmox is already installed; phase 5
(verification) always runs.

## Requirements

- Hetzner dedicated server booted into rescue mode (connect as `root`).
- Run with `become: true`.
- `ansible.posix` collection (for `authorized_key`) — already a dependency of
  the `nkg.proxmox` collection.

## Required variables

| Variable | Description |
|----------|-------------|
| `hetzner_image_disks` | List of host disk device paths to install onto, e.g. `['/dev/nvme0n1', '/dev/nvme1n1']`. **Wiped.** |
| `hetzner_image_hostname` | Proxmox node hostname. |
| `hetzner_image_root_password_hashed` | Hashed root password for the installer (`openssl passwd -6`). |
| `hetzner_image_ssh_public_key` | SSH public key for root. |
| `hetzner_image_email` | Admin contact email (installer mailto). |
| `hetzner_image_qemu.password` | Temporary root password for the QEMU staging VM (configure phase). |
| `hetzner_image_dns_servers` | DNS resolvers written inside the guest. |

Networking/install defaults you will usually want to review:
`hetzner_image_private_subnet`, `hetzner_image_hetzner_secondary_ip`,
`hetzner_image_filesystem` / `hetzner_image_zfs_raid`, `hetzner_image_qemu`
(cpus/memory/timeouts), and `hetzner_image_admin_user`.

See `defaults/main.yml` and `meta/argument_specs.yml` for the full list.

## Example Playbook

```yaml
- hosts: hetzner_servers
  become: true
  gather_facts: true
  roles:
    - role: nkg.proxmox.hetzner_image
      vars:
        hetzner_image_disks:
          - /dev/nvme0n1
          - /dev/nvme1n1
        hetzner_image_hostname: pve
        hetzner_image_domain: example.com
        hetzner_image_email: admin@example.com
        hetzner_image_root_password_hashed: "$6$..."   # openssl passwd -6
        hetzner_image_ssh_public_key: "ssh-ed25519 AAAA... admin@example"
        hetzner_image_dns_servers:
          - 185.12.64.1
          - 185.12.64.2
        hetzner_image_qemu:
          cpus: 4
          memory: 4096
          ssh_port: 5555
          install_timeout: 3600
          config_timeout: 3600
          password: "a-temporary-staging-password"
```

## Status

Ported and merged from `Regularmusic/iac` roles `proxmox_prereqs`,
`proxmox_install`, `proxmox_configure`, and `proxmox_post` (plus the
`hetzner.yml` pre_tasks and the shared `network_facts.yml` /
`ensure_admin_user.yml` includes). Environment-specific Regularmusic values
have been abstracted into role variables with sensible/empty defaults.

The original `proxmox_post` role saved the API token generated by
`first-boot.sh` into a SOPS-encrypted secrets file on the control node. **That
SOPS coupling has been dropped.** Instead, the post phase captures the token
and exposes it as the `hetzner_image_api_token` fact (and optionally writes it
to `hetzner_image_token_output_path`); pipe that into your own secret store.
Toggle with `hetzner_image_capture_token` /
`hetzner_image_remove_token_after_capture`.
